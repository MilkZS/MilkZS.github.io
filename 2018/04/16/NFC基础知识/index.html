NFC基础知识

Posted on 2018-04-16 | In NFC

NFC基础知识

1. NFC的运行模式

1.1 NFC包含三种主要的运行模式

  • R/W模式

        该模式交互是单向的。工作模式是具有NFC功能的设备（NFC Device）和NFC TAG之间进行的交互。但是只能由NFC向NFC TAG进行读取和写入操作，例如通过具有NFC功能的手机向北京公交地铁卡充值一样，只能由手机对卡进行查看余额，或者用手机对卡进行充值，但是不能由卡对手机进行读写操作。

        NFC TAG有四种类型，由不同的厂家推出。对应的四种芯片名分别是Topaz（ISO 14443
    Type A）、MIFARE（ISO14443 TypeA）、FeliCa（FeliCa）、MIFARE-DESFire（ISO
    14443 TypeA TypeB）。

  • P2P模式

        该模式交互是双向的。该模式是两个NFC Device之间进行信息的交互，互相读取写入信息。其中涉及到了几个关键词，LLC、LLCP、SNEP、CHP。LLC为P2P协议的最高层，该层使用的协议为LLCP协议，SNEP挨着LLC层，SNEP协议可以使得两个Device之间进行通信。这个过程个人理解类似于WEB中的TCP通信。

  • CE模式

        个人理解该模式主要用于停车场门卡系列，NFC位于手机的SIM卡中，用户通过接触TAG发生交互。

1.2 模式程序运行流程

  • R/W模式

        该模式下,Android平台提供了两种分发系统：TAG分发系统和前台分发系统。

    A. TAG分发系统

        当扫描到一个NFC TAG后，NFC系统模块会先尝试读取其中的内容，如果可以成功映射成Android系统直接支持的数据类型，那么NFC系统模块将会发送一个ACTION_NDEF_DISCOVERED（携带该NDEF消息和一个代表该NFC TAG的Tag对象）的Intent给那些注册了ACTION_NDEF_DISCOVERED通知感兴趣的ACtivity。如果找到了该ACtivity，则将该Intent发送给他，如果没找到，则尝试发送一个ACTION_TECH_DISCOVERED的Intent（包含一个代表目标的NFC Tag的Tag对象）。

        如果不能转换成系统直接支持的数据类型，也会发送一个ACTION_TECH_DISCOVERED的Intent（包含一个代表目标的NFC Tag的Tag对象）。如果找打了对该Intent感兴趣的ACtivity，则将Intent发送给他。如果没找到，则NFC 模块最后发送一个ACTION_TAG_DISCOVERED的Intent（包含一个Tag对象）。寻找对它感兴趣的ACtivity并将Intent发送给他。

    B. 前台分发系统

        TAG分发系统会搜寻系统内所有的满足条件的ACtivity，而前台分发系统只会寻找前台（当前显示的ACtivity）的满足条件的ACtivity。

  • p2p模式

        该模式使用了SNEP协议。个人理解，该模式重点在于发送端的处理，接收端使用TAG分发系统来接收信息并处理。

        发送端重要的主要有两个函数：

    setNdefPushMessageCallback:设置一个回调对象，如果该对象不为空，那么当前ACtivity会调用Android beam（Android 4.0以上才有）来发送所要发的内容。该函数作用是：两个设备在NFC有效的范围内时，系统会弹出数据发送通知框。如果用户选择本机发送数据，则NFC模块回调对象会获取需要发送的数据。

    setOnNdefPushCompleteCallback:当数据发送完毕后通知回调对象。当数据发送完毕后，系统会调用该回调对象的函数。

  • CE模式

        在Android平台中，需要应用程序加载一个nfc_extras,jar 的动态库才可以使用CE模式。

2. NFC系统模块

    系统模块运行在NFC.apk中。

  • NFCService

        devices启动之后的主service，用来为NFC各种功能提供服务。

  • P2pLinkManager

        该包包含三个与传输相关的server，分别为SnepServer、NdefPushServer、HandoverServer每一个Server都有回调接口类，接口类的实现在P2pLinkManager的内部类来实现。

  • HandoverServer

        负责处理NFC Connection HandoverServer协议，具体的数据传输则由P2pLinkManager来实现。Android是默认使用蓝牙来传输数据的。

  • P2pEventManager

        用来处理P2p中和用户交互相关的逻辑。

3. Google相关基本类

  • NfcManager

        一个NFC adapter的管理器，可以列出所有此android设备支持的NFC adapter.只不过大部分android 设备只有一个NFC adapter，所以你大部分情况下可以直接用静态方法 getDefaultAdapter(context)可以来获取适配器，这个适配器是全局的。

  • NfcAdapter

        表示本设备的NFC adapter,可以定义Intent来请求将系统检测到tags的提醒发送到你的Activity.并提供方法去注册前台tag提醒发布和前台NDEF推送。

    前台NDEF推送是当前android版本唯一支持的p2p NFC通信方式。

  • NdefMessage and NdefRecord

        NDEF是NFC论坛定义的数据结构，用来有效的存数据到NFC tags.比如文本，URL，和其他MIME类型。一个NdefMessage扮演一个容器，这个容器存哪些发送和读到的数据。一个NdefMessage对象包含0或多个NdefRecord,每个NDEF record有一个类型，比如文本，URL，智慧型海报/广告，或其他MIME数据。在NDEFMessage里的第一个NfcRecord的类型用来发送tag到一个android设备上的activity。

  • Tag

        标示一个被动的NFC目标，比如tag，card，钥匙挂扣，甚至是一个电话模拟的的NFC卡。当一个tag被检测到，一个tag对象将被创建并且封装到一个Intent里，然后NFC发布系统将这个Intent用startActivity发送到注册了接受这种Intent的activity里。你可以用getTechList()方法来得到这个tag支持的技术细节和创建一个android.nfc.tech提供的相应的TagTechnology对象。

4. HCE 和 SE

4.1 概念

  • HCE：基于主机的卡模拟
  • SE：安全元件的简称

4.2 背景

  • HCE

    在一部配备NFC功能的手机中实现卡模拟，目前有两种方式：

    1. 基于硬件的虚拟卡模拟（VCM）

          该模式下，需要提供安全模块SE，NFC外部读取器读取到外部指令传给SE进行处理，然后SE处
      理完毕后传递给NFC控制器。

    2. 基于软件的主机卡模拟（HCM）

          该模式下，不需要提供SE，而是由在手机中运行的一个应用或云端的服务器来完成SE的功能。此时，NFC芯片接收到的数据或由操作系统传递给手机中的应用，或通过网络发送给云端服务器来完成交互。这两种方式都绕过了SE的限制。

  • SE

    通常以芯片的方式存在，用来防止外部恶意数据，芯片中由加密和解密的逻辑电路。

4.3 应用

    Android系统上的HCE技术是通过系统服务实现的（HCE服务），使用服务的优势是它可以一直在后台运行而不需要由用户界面，这样方便了用户的使用，用户在刷门禁卡或支付时候不需要打开程序，直接将手机贴上去就可以。当然，配合上一个完美的用户界面UI可以实现更多功能，如充值，查看余额等等。

4.4 实现过程

    想用HCE实现NFC功能，首先需要创建一个HCE服务，Android4.4以上由HCE服务提供的一个非常方便的基类，我们可以通过继承基类来实现自己需要的HCE服务。对于已经存在的NFC系统，可以在HCE服务中实现NFC读卡器期望的应用程协议。

    注意：一般尽量减少NFC设备之间的数据交换量从而来保证用户不需要花大量时间将手机贴在读卡器上。
